# 第 1 章 · 概念入门：什么是 Skill

> 读完这一章你应该能：① 用一句话向同事解释清楚 Skill 是什么；② 知道它和 system prompt、tool、sub-agent 的**本质区别**；③ 判断"我现在的需求该不该用 Skill"。

---

## 1.1 官方定义

Anthropic 在 2025-10-16 发布 Skills 时给的定义：

> **"folders of instructions, scripts, and resources that Claude loads dynamically to improve performance on specialized tasks."**
>
> —— 一组包含**指令、脚本、资源**的文件夹，Claude 在需要时**动态加载**它们以提升特定任务的表现。

把这句话拆开看，三个关键词：

| 关键词 | 含义 |
|--------|------|
| **folders** | 物理形态是一个文件夹（不是单文件、不是服务、不是注册表项） |
| **instructions + scripts + resources** | 三类内容可以打包在一起 |
| **loads dynamically** | 不是"始终在"，而是"按需加载"——这是关键 |

类比一下：
- **System prompt = 员工胸卡**：永远挂在身上，每次对话都生效
- **Skill = 工具箱 + 操作手册**：放在工位上，需要时去翻
- **MCP = 外部供应商对接系统**：把外部数据/工具接到公司里
- **Sub-agent = 另一个专家同事**：独立的脑子，可以独立思考

---

## 1.2 三个例子直观感受

### 例子 1：PDF 表单填写

**没有 Skill 的做法**：
- 用户上传 PDF 表格："帮我填上姓名和地址"
- Claude 看得懂，但**不知道怎么操作 PDF 表单字段**（PDF 表单字段是 AcroForm/XFA 结构，不是普通文本）

**有 Skill 之后**：
- 装一个 `pdf-form-filler` skill
- SKILL.md 里写："遇到 PDF 表单，先运行 `extract_fields.py` 提取字段，问用户填什么值，再生成新 PDF"
- 用户提需求 → Claude 看到 description 匹配 → 读 SKILL.md → 跑脚本 → 完成

**关键差别**：Skill 把"**操作流程**"和"**专业工具**"打包在一起了。

### 例子 2：代码审查

**没有 Skill**：
- 用户每次都要粘贴："帮我审查代码，要看安全性、命名规范、测试覆盖……"
- 换个项目，规范又不一样，之前的指令又得重写

**有 Skill**：
- 装一个 `code-review` skill，团队共享
- 团队规范一次写好，永远生效
- 任何人在任何项目下用 Claude Code，描述自动匹配就会触发

### 例子 3：会议纪要（中文场景）

- 装一个 `meeting-minutes` skill
- 用户说"整理一下这段会议录音" → 触发
- 内部有 `references/weekly.md` `references/retro.md` `references/client.md` 三种模板
- Claude 根据会议类型**按需读对应的 reference**，输出对应格式

## 1.3 Skill 的核心机制：三层渐进式披露

这是理解 Skill 最关键的一张图：

```
┌────────────────────────────────────────────┐
│  L1：元数据层                              │
│  ─────────────────                         │
│  所有 Skill 的 name + description          │
│  启动时一次性注入到 system prompt           │
│  约 100 词 / Skill                          │
│  Claude 用来"知道有什么可用的"             │
└────────────────────────────────────────────┘
                    ↓ 任务匹配
┌────────────────────────────────────────────┐
│  L2：主体层                                │
│  ──────────                                │
│  匹配的 Skill 的 SKILL.md 完整 Markdown    │
│  包含工作流程、检查点、调用约定             │
│  建议 < 500 行                              │
│  Claude 用来"知道怎么做"                   │
└────────────────────────────────────────────┘
                    ↓ 进一步需要
┌────────────────────────────────────────────┐
│  L3+：资源层                               │
│  ──────────────                            │
│  scripts/、references/、assets/            │
│  按需读取：Claude 用 Read/Bash 工具读取    │
│  大小无限制                                │
│  Claude 用来"拿到执行能力和详细知识"       │
└────────────────────────────────────────────┘
```

**这套设计解决什么痛点**：

- **痛点 1：上下文窗口有限**——如果每次把所有 skill 内容都塞进去，10 个 skill 就撑爆窗口
- **痛点 2：行为一致性**——每次对话都要重复粘贴"按我们的规范做"，不可持续
- **痛点 3：专业能力沉淀**——团队里有人搞懂的事，沉淀成 skill，让别人也能用

## 1.4 跟四个容易混淆概念的边界

| 维度 | System Prompt | Tool (Function Call) | Skill | Sub-agent |
|------|--------------|----------------------|-------|-----------|
| **本质** | 始终在胸口的"工作要求" | 可调用的函数 | 按需加载的操作手册 | 独立运行的 LLM 实例 |
| **加载时机** | 始终加载 | 每次请求可调用 | 描述匹配时加载 | 显式启动 |
| **状态** | 无状态 | 函数调用 | 无状态（脚本纯函数） | 有独立上下文 |
| **是否需要 LLM** | 否 | 否 | 否（脚本不需要 LLM） | 是 |
| **Token 占用** | 常驻 | 工具描述常驻 | 只 description 常驻 | 独立一份上下文 |
| **典型用法** | 项目基础规范 | 调 API、查数据库 | 业务 SOP、领域知识 | 隔离任务、深度推理 |
| **比喻** | 胸卡/员工手册 | 扳手/工具 | 工位上的工具箱+操作手册 | 隔壁工位的同事 |

### 关键边界判断

**什么时候用 System Prompt**：
- 项目级高频规则（"我们用 2 空格缩进"）
- 简短固定的角色定义（"你是高级 Python 工程师"）
- ≤ 几百 token 的硬性要求

**什么时候用 Tool / Function Call**：
- 需要**精确、可重复**的外部操作（调 API、查 DB、发邮件）
- 单个动作，不是流程
- 已有现成 API

**什么时候用 Skill**：
- 一段**完整的工作流程**（多步骤 + 决策点）
- 团队/项目**专业知识沉淀**
- 内容 > 100 行，**不适合放 system prompt**
- 需要**带附件**（脚本、模板、参考文档）

**什么时候用 Sub-agent**：
- 任务需要**隔离上下文**（不让主对话被污染）
- 任务**可以并行**（多个子 agent 同时干不同事）
- 需要**独立推理**（不是 SOP 流程，是要思考的）

### Skill 和 Sub-agent 的对照（重点澄清）

| 维度 | Skill | Sub-agent |
|------|-------|-----------|
| 本质 | 函数/工具 | LLM 智能体 |
| 是否需要 LLM 推理 | ❌ | ✅ |
| 是否有状态 | ❌ | ✅ |
| 能否自主决策 | ❌ | ✅ |
| 典型用途 | 数据处理、规则匹配 | 复杂分析、专业咨询 |

**一句话总结**：Skill 是"无脑的执行者"，Sub-agent 是"有脑的思考者"。Skill 适合 SOP 化的工作，Sub-agent 适合需要推理的任务。

## 1.5 我该不该用 Skill？决策树

```
开始
 ↓
这个任务是固定的、重复的工作流程吗？
 ├─ 否 → 用 Prompt / 对话
 └─ 是 ↓
    这个流程能用"自然语言 + 偶尔调脚本"表达吗？
     ├─ 否 → 用 Sub-agent（让另一个 LLM 想）
     └─ 是 ↓
        内容 ≤ 100 行？
         ├─ 是 → 放 system prompt
         └─ 否 ↓
            需要附带脚本/模板/参考文档吗？
             ├─ 否 → 放 system prompt（如果还是塞得下）
             └─ 是 → ✅ 用 Skill
```

实际经验：
- **"**用户每次让我做的事**"→ 流程化 + 需要附件 → 写 Skill
- "**用户偶尔让我做一次**"→ 直接对话 / 写命令
- "**任务每次都不同、要临场思考**"→ Sub-agent 或直接对话

## 1.6 Skill 的"杀手特性"

不是所有优势都重要，这四个是真正改变工作方式的：

### 1. 按需加载 → token 经济性

装 100 个 skill，**常驻上下文**只占 100 × 100 词 = 1 万词。一次对话只把用到的那个 skill 主体拉进去。

### 2. 跨平台可移植

同一个 skill 目录：
- 复制到 `~/.claude/skills/` → Claude Code 立即可用
- 上传到 Claude.ai → 网页版也能用
- 通过 API 上传 → 任何集成 Anthropic 的应用都能用

**一次编写，多处运行**。

### 3. 可组合

可以同时装多个 skill：
- 装 `pdf-skill` + `code-review-skill` + `git-commit-skill`
- 一次任务可能触发多个（比如改完代码 → 触发 code-review 审查 → 触发 git-commit 提交）
- Claude 会自动协调

### 4. 可执行代码

Skill 不只是文档，可以**带脚本**：
- 需要确定性计算？（如 PDF 字段提取）→ 用 Python 脚本
- 需要执行系统命令？（如 git 操作）→ 用 Shell 脚本
- 脚本是**纯函数式**的：入参 → 出参，不感知对话上下文

---

## 1.7 一句话总结

> **Skill = 一份带元数据的 SOP（标准操作流程）+ 工具箱**，Claude 启动时只看见名字和简介，需要时再翻内容。
>
> 它**不是插件**（不是注册到系统的二进制）、**不是 API**（不是网络服务）、**不是 prompt**（不是单段提示词）。
>
> 它就是**一个文件夹**，里面是 `SKILL.md` + 可选附件。
